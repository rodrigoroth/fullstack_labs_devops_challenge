# FullStack Labs DevOps Challenge

## About the Challenge

This project was developed as part of the FullStack Labs DevOps Challenge. The goal is to design and implement a CI/CD pipeline and an Infrastructure as Code (IaC) solution using Terraform and AWS.

The requirements include:

- Deploying a static application hosted on S3 behind a CloudFront distribution.
- Storing CloudFront access logs in an S3 bucket.
- Implementing a secure and scalable infrastructure following DevOps best practices.
- Managing different isolated environments (devel, stage, and prod).
- Enforcing a strict branching strategy using GitHub Actions.

## Infrastructure Overview (Terraform)

The Terraform configuration provisions the following resources:

- **S3 Bucket for Static Website**
  - Hosts the website files (HTML, CSS, JS).
  - Public access fully blocked.
  - Only accessible through CloudFront (via Origin Access Identity).

- **S3 Bucket for CloudFront Logs**
  - Stores access logs generated by CloudFront.

- **CloudFront Distribution**
  - Serves the static website securely over HTTPS.
  - Redirects HTTP to HTTPS.
  - Uses an Origin Access Identity (OAI) to securely access the S3 bucket.

- **IAM Policies**
  - Allow CloudFront to write logs to the log bucket.
  - Allow OAI to read files from the static site bucket.

- **Remote Terraform Backend**
  - Stores Terraform state files in an S3 bucket.
  - Supports multiple environments using Terraform workspaces (devel, stage, prod).

## Branching Strategy

The repository uses a strict branching model with the following branches:

- `devel` → Development environment.
- `stage` → Staging environment.
- `prod` → Production environment.

### Branch Rules

- Pull requests from `feature/*` or `bugfix/*` can only be merged into `devel`.
- `stage` only accepts pull requests coming from `devel`.
- `prod` only accepts pull requests coming from `stage`.
- Direct pushes to `devel`, `stage`, and `prod` are blocked.

## Enforcing Branch Rules (GitHub Actions)

A GitHub Actions workflow enforces the branch strategy by validating pull requests:

- Checks if the source and target branches follow the required flow.
- Blocks any pull request that violates the branching strategy.

## CI/CD Pipeline (GitHub Actions)

The CI pipeline is triggered on every pull request or push to the main branches (`devel`, `stage`, `prod`).

### Pipeline Steps

1. Install dependencies (`npm install`).
2. Run linter (`npm run lint`).
3. Check code formatting (`npm run prettier --check .`).
4. Run tests (`npm run test`).
5. Build the application (`npm run build`).
